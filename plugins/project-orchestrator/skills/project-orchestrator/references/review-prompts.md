# レビュープロンプトライブラリ

各フェーズに適したドメインを選択する。フェーズが複数の領域にまたがる場合はドメインを組み合わせる。

---

## データベース / スキーマ変更

### 専門家
> データベース設計の専門家として、フェーズ {N} のスキーマ変更をレビューせよ:
> - 制約（UNIQUE、FK、NOT NULL）は正しく定義されているか？
> - 頻繁にクエリされるカラムにインデックスが配置されているか？
> - マイグレーションは既存データとの後方互換性があるか？
> - ポリモーフィックリレーションは型安全か？
> - パフォーマンス上の懸念はないか？（フルテーブルスキャン、インデックス不足）

### 批判的
> スキーマ変更のデータ整合性リスクを批判的にレビューせよ:
> - 既存データが新しい制約に違反する可能性はないか？
> - マイグレーション後に孤立レコードが発生しないか？
> - 同時書き込みが制約違反を引き起こす可能性はないか？
> - ロールバックマイグレーションは提供されているか？

---

## API / バックエンド変更

### 専門家
> バックエンド API の専門家として、フェーズ {N} の API 変更をレビューせよ:
> - レスポンス形状はフロントエンドの期待に一致しているか？
> - 保護されたエンドポイント全てに認証/認可チェックがあるか？
> - クエリは最適化されているか？（N+1 問題なし、JOIN の適切な使用）
> - エラーハンドリングは一貫しているか？（適切な HTTP ステータスコード、サイレントキャッチなし）
> - リストエンドポイントにレート制限やページネーションが適用されているか？

### 批判的
> API 変更のセキュリティと正確性を批判的にレビューせよ:
> - 未認証ユーザーが保護されたデータにアクセスできないか？
> - パブリックエンドポイントが機密フィールド（メール、トークン、パスワード）を漏洩していないか？
> - 論理削除済みまたは非公開のレコードがパブリックレスポンスから除外されているか？
> - 悪意のある入力が SQL インジェクションや OWASP Top 10 の問題を引き起こす可能性はないか？
> - 書き込み操作に競合状態はないか？

---

## フロントエンド / コンポーネント変更

### 専門家
> フロントエンドの専門家として、フェーズ {N} のコンポーネント変更をレビューせよ:
> - コンポーネントはプロジェクトの命名規則とファイル構成に従っているか？
> - props/emits のインターフェースは正しく型付けされているか？
> - データフェッチは適切に処理されているか？（ローディング状態、エラー状態）
> - auto-import の名前解決は正しいか？（二重プレフィックスなし）
> - SSR 互換性は維持されているか？（setup 内で window/document を使用していないか）

### 批判的
> フロントエンド変更のリグレッションを批判的にレビューせよ:
> - テンプレートで参照されているが存在しないコンポーネントはないか？
> - useFetch の API エンドポイントパスは実際のサーバールートと一致しているか？
> - 全ての保護ページに認証ガードが設定されているか？
> - ユーザーに表示されるデータは適切にエスケープされているか？（XSS 防止）
> - オリジナル/デザインと比較して欠落している UI 要素はないか？

---

## インフラ / 設定変更

### 専門家
> インフラの専門家として、フェーズ {N} の設定変更をレビューせよ:
> - 環境変数は文書化され、正しく参照されているか？
> - シークレットはバージョン管理と公開設定から除外されているか？
> - デプロイ設定はターゲットプラットフォームに対して正しいか？
> - キャッシュ戦略は適切か？
> - リソース制限はプラットフォームの制約内か？

### 批判的
> 設定変更のデプロイリスクを批判的にレビューせよ:
> - プレースホルダ値（TODO、xxx、example.com）が残っていないか？
> - 設定ミスがデータ損失やセキュリティ露出を引き起こす可能性はないか？
> - 設定が本番環境を壊した場合のロールバックパスはあるか？
> - OAuth リダイレクト URL は正しいドメインに設定されているか？

---

## テスト変更

### 専門家
> テストの専門家として、フェーズ {N} のテスト変更をレビューせよ:
> - テストは意味のある振る舞いを検証しているか？（単なるスナップショット一致ではなく）
> - エッジケースはカバーされているか？（空データ、不正入力、認証境界）
> - モックは適切か、統合テストを使うべきか？
> - テストは決定的か？（タイミング依存なし、外部呼び出しなし）
> - テスト実行時間は妥当か？

### 批判的
> テストカバレッジの欠落を批判的にレビューせよ:
> - テストカバレッジのない重要なパスはないか？
> - 機能が壊れていてもテストが通る可能性はないか？（偽陽性）
> - セキュリティ境界はテストされているか？（未認証アクセス試行）
> - テストはユーザー間のデータ分離を検証しているか？

---

## 汎用（技術非依存）

### 専門家
> シニアエンジニアとして、フェーズ {N} の変更をレビューせよ:
> - 実装はフェーズドキュメントの目的に合致しているか？
> - 同じ結果を達成するよりシンプルなアプローチはないか？
> - コードはプロジェクトの既存パターンと一貫しているか？
> - エラーケースは明示的に処理されているか？（サイレントフェイルなし）

### 批判的
> フェーズ {N} の完全性とリスクを批判的にレビューせよ:
> - フェーズドキュメントの全タスクが対応されているか？
> - これらの変更が後続フェーズで問題を引き起こす可能性はないか？
> - 検証すべき仮定はないか？
> - ロールバック手順はテスト済みで文書化されているか？
