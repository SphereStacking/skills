# テスト実装ガイド

テスト実装タスクを委譲する際に、implementer 起動プロンプトに注入するガイド。
t-wada 流 TDD の思想を基盤とする。

---

## 基本原則

- **テストは仕様書** — テストコードを読めば振る舞いが分かる状態を目指す
- **動作するきれいなコード** — Red→Green→Refactor のサイクルで小さく確実に進める
- **テストコードもプロダクションコードと同じ品質基準** で書く

## サイクル

1. **Red** — まず失敗するテストを書く。テストが仕様を表現する
2. **Green** — 最小限の実装でテストを通す。過剰な実装をしない
3. **Refactor** — テストが通った状態を維持しつつコードを整理する

小さいステップで回す。一度に大きく進めない。

## テスト設計

### ケース設計の順序
1. 正常系（最も基本的な成功パス）
2. 境界値（0、1、上限、空文字、null）
3. 異常系（不正入力、権限なし、リソース不在）
4. エッジケース（同時実行、大量データ、特殊文字）

### 三角測量
1つのテストで自信が持てないなら、別の角度からもう1つテストを追加する。
ただし冗長なテストは避ける — 新しい情報を与えないテストは書かない。

### テスト名
「何の条件で、何をしたら、どうなるか」が読み取れる命名にする。

```
// 良い例
"イベントが存在しない場合 404 を返す"
"未認証ユーザーのリクエストを拒否する"

// 悪い例
"test1"
"エラーケース"
```

## テスト構造

### Arrange-Act-Assert
各テストで3フェーズを明確に分離する:

```
// Arrange — テストデータとセットアップ
// Act — テスト対象の実行（1つだけ）
// Assert — 結果の検証
```

### 1テスト1振る舞い
1つのテストケースで検証する振る舞いは1つだけ。
複数の assert があっても、全て同じ振る舞いの異なる側面であること。

### 独立性
- テストの実行順序に依存しない
- テスト間で状態を共有しない
- 各テストが独立してセットアップ・クリーンアップする

## モックの方針

- モックは最小限にする
- 実際の動作に近いテストほど価値が高い
- モックが増えたら設計を疑う — テスト対象の責務が大きすぎる兆候
- 外部 API、時刻、ランダム値はモック対象として妥当

## やらないこと

- テスト対象のコードを変更しない（バグを見つけたらレポートに記載のみ）
- 実装の詳細に依存するテストを書かない（private メソッドを直接テストしない）
- フレイキーテストを書くくらいならテストしない
- カバレッジ100%を目的にしない — 意味のあるテストだけを書く
