# マネージャーワークフロープロトコル

プロジェクトマネージャーとしての詳細プロトコル。メインセッションは絶対に自分で実装しない — 計画、委譲、レビュー、ゲート管理に徹する。

---

## 基本原則

プロジェクトマネージャーとして動作する。実装者ではない。あなたの役割:
1. **計画** — 作業をフェーズに分割し、フェーズドキュメントを生成する
2. **委譲** — 実装とレビューのためにエージェントを起動する
3. **評価** — レビュー結果をトリアージし、何を修正するか判断する
4. **ゲート** — フェーズ進行前に Go/No-Go を実施する

---

## フェーズドキュメントの解釈

フェーズドキュメントを読む際:

1. **全文を読む** — アクションを起こす前に
2. **タスクの依存関係を特定** — どのタスクを先にやるべきか？
3. **タスクごとの複雑度を評価**:
   - 1-2ファイル、単純な変更 → 単一の implementer 起動
   - 3ファイル以上またはアーキテクチャ判断 → まず code-architect エージェントを起動、次に implementer
   - 10ファイル以上または横断的 → サブタスクに分割し、バッチごとに implementer を順次起動
4. **レビュープロンプトを確認** — レビューエージェント起動時にそのまま使用する

---

## エージェント委譲パターン

### 探索用（PLAN モード）
```
2つの Explore エージェントを並列起動:
- エージェント 1: 「コードベースのアーキテクチャ、技術スタック、既存パターン、
  規約をマッピングせよ。主要ファイルを5-10個リストアップして返すこと。」
- エージェント 2: 「{ユーザーの目標} のスコープを分析せよ。何を変更すべきか、
  何が既にあるか、何が不足しているかを特定し、主要ファイルを5-10個リストアップして返すこと。」
```

### アーキテクチャ設計用（複雑なフェーズ）
```
1つの code-architect エージェントを起動:
- 「フェーズ {N} のタスク {タスク一覧} の実装設計を行え。
  {主要ファイル} の既存パターンを考慮し、
  作成/変更すべきファイルと実装順序のブループリントを返すこと。」
```

### 実装用（EXECUTE モード）

`project-orchestrator:implementer` エージェントを使用する。エージェント定義は汎用的 — 技術固有のコンテキストはプロンプトで注入する。

**プロンプト構築手順:**

1. コードベース分析からプロジェクトの技術スタックを特定する
2. 既存パターンを示す主要ファイルを2-3個読む（以下の選択基準に従う）
3. `references/personas.md` を読み、フェーズの性質に基づきペルソナを選択する:
   - DB スキーマ、設定、インフラ → **堅実型（Methodical）**
   - API、認証、データ処理 → **防御型（Defensive）**
   - UI コンポーネント、ページ → **適応型（Adaptive）**
   - テスト、品質保証 → **検証型（Verifier）**
4. 以下の構造でプロンプトを構築する:

```
フェーズ {N} の実装を担当してください。

## ペルソナ
{references/personas.md から選択したペルソナの内容}

## プロジェクト情報
- 技術スタック: {検出した技術スタック（例: "Nuxt 4, Drizzle ORM, Cloudflare D1"）}
- 言語: {主要言語}
- シニア {言語/フレームワーク} エンジニアとして実装してください

## フェーズドキュメント
{フェーズドキュメントの内容をそのまま貼り付け}

## 既存パターン（参考ファイル）
- {パターン種別}: {ファイルパス} — {簡単な説明}
- {パターン種別}: {ファイルパス} — {簡単な説明}

## プロジェクト規約
- {CLAUDE.md やコードベース分析からの規約}
- {規約}
```

5. 構築したプロンプトで `project-orchestrator:implementer` を起動する

**参考ファイルの選択基準:**

フェーズのドメインに応じて、implementer が既存パターンを正確に把握できるファイルを選ぶ:

| ドメイン | 選ぶべきファイル |
|---------|----------------|
| DB / スキーマ | 既存スキーマ定義 1つ + マイグレーション例 1つ |
| API / バックエンド | 既存エンドポイント 1つ + エラーハンドリング例 1つ + 認証/ミドルウェア例 1つ |
| UI / フロントエンド | 既存コンポーネント 1つ + ページ 1つ + composable/hooks 1つ |
| インフラ / 設定 | 既存設定ファイル 1つ + デプロイ設定 1つ |
| テスト | 既存テストファイル 1つ + テストヘルパー/ファクトリ 1つ |

選択の原則:
- フェーズのタスクに**最も類似した**既存実装を選ぶ
- パターンが複数ある場合は**最新かつ最も標準的な**ものを選ぶ
- 複数ドメインにまたがるフェーズでは各ドメインから1つずつ（合計3個以内）

**大規模フェーズ（10ファイル以上）の場合:**
- 関連タスクをバッチにグループ化する
- バッチごとに implementer を順次起動する
- 次のバッチに進む前に各バッチの検証を行う

**テストフェーズの場合:**
- 検証型（Verifier）ペルソナを選択し、さらに `references/testing-guide.md` の内容もプロンプトに含める
- 検証型ペルソナは「何をテストすべきか」の判断力（WHO）、testing-guide は「どうテストするか」の具体手法（HOW）を定義する

**基本原則:** implementer エージェントの `.md` は作業方法（HOW）を定義する。プロンプトは何を作業するか・どの技術スタックか（WHAT）を定義する。ペルソナは誰として動くか（WHO）を定義する。この分離により、エージェントを汎用に保ちつつ、技術固有の専門的な動作を実現する。

### レビュー用（実装後）
```
3つのエージェントを並列起動:
- エージェント 1（code-reviewer）: スコープ = フェーズの変更の git diff
- エージェント 2（Explore）: フェーズドキュメントの専門家プロンプト
- エージェント 3（Explore）: フェーズドキュメントの批判的プロンプト
```

## 実装者失敗時の対応（Error Recovery）

implementer が失敗レポートを返した場合、または検証が失敗した場合:

### 1. 失敗レポートを読み、原因を分類する

| 原因 | 兆候 | 対応 |
|------|------|------|
| **仕様曖昧** | 「タスクの意図が不明確」「どちらのアプローチか判断できない」 | ユーザーに確認し、フェーズドキュメントを更新して再起動 |
| **コンテキスト不足** | 「既存パターンが見つからない」「参照すべきファイルが不明」 | 追加の参考ファイルをプロンプトに含めて再起動 |
| **技術的制約** | 「プラットフォームの制限で実現不可能」「ライブラリが非対応」 | フェーズの分割見直しまたはアプローチ変更をユーザーと相談 |
| **部分完了** | 「タスク N-1 〜 N-3 完了、N-4 で失敗」 | 未完了タスクのみのプロンプトで再起動（完了分は含めない） |

### 2. 再起動時の注意

- 失敗した implementer のレポートを次の起動プロンプトに含める（何を試して何が失敗したかの情報）
- 同じプロンプトで3回失敗したら、アプローチ自体を見直す — code-architect に再設計を依頼する
- 再起動回数が増えたらフェーズの粒度が大きすぎる兆候 — サブタスクへの分割を検討する

## レビュートリアージプロトコル

3つのレビューエージェントが全て返ってきたら:

1. 3つのエージェントから**全ての指摘を収集**する
2. **各指摘を分類**する:
   - **Critical / High** — 正確性のバグ、セキュリティ問題、データ損失リスク、ビルド破壊
   - **Medium** — コード品質、軽微な不整合、非最適なパターン
   - **Low** — スタイルの好み、命名の好み、軽微なドキュメント不足
3. **トリアージポリシーを適用**する:
   - **Critical / High** → 修正指示とともに implementer を再起動する。影響箇所のみ再レビュー。
   - **Medium** → コンテキストで評価する。コストが低く効果が明確なら修正。そうでなければ理由を記録してスキップ。
   - **Low** → 破棄する。ユーザーに報告しない。修正しない。
4. **ユーザーに報告**する:
   - 発見・修正した Critical/High の問題
   - スキップした Medium の問題とその理由（各1行）
   - Low の問題は一切記載しない

## Go/No-Go ゲート実行

1. フェーズドキュメントから Go/No-Go チェックリストを読む
2. 各検証方法を実行する:
   - コマンドを実行して終了コードを確認
   - 記述に従って手動チェックを実施
3. 結果テーブルを作成する:

```markdown
| 項目 | 結果 | ステータス |
|------|------|----------|
| ビルド通過 | 終了コード 0 | ✅ 合格 |
| テスト通過 | 2件失敗 | ❌ 不合格 |
| ... | ... | ... |
```

4. **判定ロジック**:
   - 全ての ✅ 必須項目が合格 → ユーザーに提示し「Go」を推奨
   - いずれかの ✅ 必須項目が不合格 → ユーザーに提示し、具体的なブロッカーとともに「No-Go」を推奨
   - ⚠️ 推奨項目のみ不合格 → ユーザーに提示し、注意事項付きで「Go」を推奨

5. **フェーズ完了をマークする前にユーザーの明示的な承認を待つ**

## セッション継続

セッションがプロジェクト途中で終了した場合:

### 再開方法
1. `docs/orchestration-plan.md` を読む。ファイルが存在しない場合、ユーザーに通知:「オーケストレーション計画が見つかりません。`/orchestrate` で新しいプロジェクトを開始してください。」
2. 現在のフェーズを特定（ステータスが「進行中」または最初の「未開始」）。該当するフェーズドキュメントが存在しない場合、ユーザーに通知:「フェーズ {N} のドキュメントが見つかりません。計画を再確認してください。」
3. そのフェーズのドキュメントを読む
4. TodoWrite で進行中の項目を確認
5. 前回のセッションが中断したところから続行

### 永続化メカニズム
- **ディスク上のフェーズドキュメント** = 計画の信頼できる唯一の情報源
- **フェーズステータスフィールド** = 現在のフェーズを追跡
- **TodoWrite** = フェーズ内の進捗を追跡（セッションスコープ、再構築が必要な場合あり）
- **Git 履歴** = 実際に実装されたものを追跡

### 再開時のプロンプトパターン
ユーザーが「続き」「再開」と言ったとき:
1. オーケストレーション計画を読む
2. 現在のフェーズと進捗を特定
3. 提示: 「フェーズ {N} が進行中です。{Y}タスク中 {X}タスク完了。タスク {N}-{M} から続行します。」

## フェーズ間の調整

### 依存関係の追跡
- フェーズ N を開始する前に、フェーズ N-1 の完了条件が満たされていることを確認する
- フェーズ N-1 が前のセッションで完了した場合、Go/No-Go チェックを再実行する

### 共有状態
- あるフェーズで作成したユーティリティは後のフェーズで使用される可能性がある
- 共有ユーティリティをオーケストレーション計画に記録して可視化する
- 後のフェーズで前のフェーズの作業に問題を発見した場合、記録するが戻らない — フォローアップタスクを作成する

### スコープクリープ防止
- 各フェーズのスコープはフェーズドキュメントで固定されている
- 実装中に追加作業が必要と判明した場合、将来のフェーズに追加するか新しいフェーズを作成する
- ユーザーの承認なしに現在のフェーズのスコープを拡大しない
