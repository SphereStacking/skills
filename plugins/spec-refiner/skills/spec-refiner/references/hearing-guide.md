# ヒアリングガイド

ユーザーの「やりたいこと」を引き出すための質問レパートリー。
これはチェックリストではない。対話の流れに合わせて、必要な質問を自然に選ぶための引き出し。

---

## 質問の原則

1. **聞き役に徹する**: 自分の意見や提案を挟まない。ユーザーの言葉を引き出すことに集中する
2. **ユーザーの言葉を使う**: 専門用語に置き換えず、ユーザーが使った表現をそのまま拾う
3. **1回に 1〜2 問**: 会話のリズムを崩さない。面接ではなく雑談のように
4. **「なぜ」を大切に**: 機能の「何」よりも「なぜそれが必要か」を聞く。動機がわかれば代替案も見える
5. **具体例を求める**: 抽象的な話が続いたら「例えばどんな場面ですか？」と具体に引き戻す
6. **絵を描かせる**: 「理想の状態を想像すると、どんな感じですか？」とビジョンを引き出す

---

## 切り口別の質問レパートリー

### きっかけ・動機

ユーザーが「何を作りたいか」だけ話している場合、背景にある動機を探る。

- 「これを作ろうと思ったきっかけは何ですか？」
- 「今はどうやってそれをやっていますか？」
- 「今のやり方で一番困っていることは？」
- 「これがないと誰が困りますか？」
- 「もしこれが完成したら、一番嬉しいのは誰ですか？」

### ユーザー像

誰のためのものかを掘り下げる。

- 「これを使う人はどんな人ですか？」
- 「使う人は技術に詳しいですか？」
- 「使う人はどのくらいの頻度で使いますか？」
- 「使う人が一番やりたいことは何ですか？」
- 「使う人が絶対にやりたくないこと（面倒、ストレス）は？」

### やりたいこと（機能面）

具体的にどんなことができるようにしたいかを聞く。

- 「ユーザーがまず最初にやることは何ですか？」
- 「その次に何をしますか？」
- 「最終的なゴール（ユーザーが達成したい状態）は？」
- 「それは毎回同じ流れですか？場合によって違う？」
- 「似たようなサービスや機能で『これはいいな』と思ったものはありますか？」
- 「逆に『これはイヤだな』と思った経験は？」

### 見た目・体験のイメージ

ユーザーの頭の中にある完成イメージを引き出す。

- 「画面のイメージはありますか？ざっくりでいいので」
- 「Web ですか？アプリですか？コマンドラインですか？」
- 「参考にしたい UI やサービスはありますか？」
- 「シンプルにしたいですか？多機能にしたいですか？」

### スコープ・優先度

何を含めて何を含めないかを探る。ただし「スコープを決めてください」と直接聞かない。

- 「最低限これだけは動いてほしい、というのはどの部分ですか？」
- 「逆に『これは後でいい』というものはありますか？」
- 「全部一気に作りますか？段階的に？」
- 「最初のバージョンで一番大事なことは？」

### 既存の文脈

ユーザーが既に持っている情報やリソースを掘り起こす。

- 「既にあるコードやシステムとの関係はありますか？」
- 「関連するドキュメントやメモはありますか？」
- 「過去に似たようなことを試みたことはありますか？」
- 「一緒に作る人はいますか？」
- 「使っている技術やツールで制約はありますか？」

### 開発スタイル・規約

プロジェクトの開発規約やこだわりを探る。**コードベース分析（`context.md`）の結果を元に、検出されたパターンを提示して確認する形が基本**。新規プロジェクトの場合のみゼロから聞く。

**既存プロジェクトの場合（提案型）:**
- 「テストは `{検出されたパターン}` に置いていますね。この方針で続けますか？」
- 「ディレクトリは `{検出されたパターン}` で整理されていますね。新機能もこれに合わせますか？」
- 「`{検出されたライブラリ}` を使っていますが、今回も使いますか？」
- 「コードベースを見る限り、`{検出された規約}` というルールがありそうですが、意識的にやっていますか？」

**新規プロジェクトの場合（ベストプラクティス提案型）:**
- 「テストファイルはソースと同じディレクトリに置くのが一般的です（`login.ts` の隣に `login.test.ts`）。この方針でいいですか？」
- 「ディレクトリは機能別（`features/auth/`, `features/payments/`）で整理すると保守しやすいですが、こうしますか？」
- 「1ファイル1責務で、200行を目安に分割するのがおすすめですが、この粒度感でいいですか？」
- 「ライブラリは車輪の再発明を避けて既存のものを積極的に使う方針でいきますか？」
- 「チームで開発する場合、ESLint / Prettier 等のフォーマッターを入れておくと統一しやすいですが、どうしますか？」

### 不安・懸念

ユーザーが心配していることを聞き出す。

- 「この計画で一番心配なことは？」
- 「技術的に不安な部分はありますか？」
- 「時間やリソースの制約はありますか？」
- 「うまくいかなかったらどうなりますか？」

### 拡張・将来

今回のスコープの先にあるビジョンを探る（ただし深入りしない）。

- 「将来的にはどこまで広げたいですか？」
- 「ユーザーが増えたらどうなりますか？」
- 「他のシステムと繋がる可能性はありますか？」

## 深掘りテクニック

ユーザーの回答をさらに掘り下げるための技法。

### オウム返し + 疑問
ユーザーの発言をそのまま繰り返し、疑問を付ける。
- ユーザー: 「通知が欲しい」
- あなた: 「通知ですね。それはどんなタイミングで届くといいですか？」

### 具体化
抽象的な発言を具体的なシナリオに落とす。
- ユーザー: 「使いやすくしたい」
- あなた: 「例えば、今のやり方で『使いにくい』と感じる具体的な場面はありますか？」

### 対比
反対のケースを聞いて境界を明確にする。
- ユーザー: 「管理者が設定できるようにしたい」
- あなた: 「一般のユーザーは設定を変えられない、ということですか？」

### ストーリー化
一連の流れとして語ってもらう。
- 「朝、ユーザーがこのシステムを開いてから、やりたいことを終えるまでの流れを想像すると、どんな感じですか？」

### 優先順位の引き出し
直接「優先度は？」と聞かない。代わりに:
- 「もし 1 つだけ選ぶとしたら、どれが一番大事ですか？」
- 「最初の 1 週間で動いてほしいのはどの部分ですか？」
