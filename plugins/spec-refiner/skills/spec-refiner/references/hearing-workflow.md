# ヒアリングワークフロー

spec-refiner の対話の進め方と、記録の更新プロセス。

---

## 対話サイクル

```
ユーザーの発言 → 言語化して確認 → 自然に次の質問 → hearing.md に蓄積
```

これを**ユーザーが終了を告げるまで**繰り返す。

---

## 対話のフロー

### 1. 受け取る

ユーザーの発言を注意深く聞く。以下の情報を拾う:

- **事実**: 具体的な要件、仕様、制約
- **意図**: なぜそうしたいのか、背景にある目的
- **感情**: こだわり、不安、期待、苛立ち
- **言外の意味**: 明示されていないが暗黙に前提としていること

### 2. 言語化する

ユーザーの発言を自分の言葉で言い換え、確認する。

**良い例:**
- 「つまり、管理者だけが設定を変更できて、一般ユーザーは閲覧のみ、ということですね」
- 「今の話をまとめると、まず X があって、それをトリガーに Y が起こる、という流れですか」

**悪い例:**
- 「承知しました」（言語化していない）
- 「つまり RBAC ベースのアクセス制御が必要ですね」（ユーザーの言葉でなく専門用語に置き換えている）

### 3. 記録する

`hearing.md` の適切なセクションに情報を追記する。

- 新しいトピックが出たら「やりたいことの詳細」にサブセクションを追加
- 既存のトピックの深掘りならそのサブセクションに追記
- どこにも当てはまらなければ「メモ」に入れて後で整理

### 4. 次の質問を選ぶ

**最優先: ユーザーの直前の発言から自然に派生する質問**

ユーザーが「通知機能が欲しい」と言ったら:
- 「どんなタイミングで通知が届くといいですか？」 ← 自然な派生
- ~~「データモデルはどうしますか？」~~ ← 脈絡がない

**次点: 気になっているが聞けていないこと**

対話が一区切りついた（ユーザーが「そんな感じかな」等と言った）場合:
- `hearing-guide.md` の質問レパートリーから、まだ触れていない切り口の質問を選ぶ
- ただし「次はこのテーマについて聞きます」とは言わない。自然に話題を振る

---

## サマリーのタイミング

### 10 回以上のやりとりが経過したら

一度立ち止まって、ここまでの内容を整理して提示する:

「ここまでの話を一度整理させてください:

- {要点 1}
- {要点 2}
- {要点 3}
...

この理解で合っていますか？抜けている部分や、違う部分があれば教えてください。」

これにより:
- ユーザーが自分の発言を俯瞰できる
- 認識のズレを早期に発見できる
- ユーザーが「あ、そういえばこれも」と新しい情報を出すきっかけになる

### サマリー後

ユーザーの確認/修正を反映し、ヒアリングを続行する。

---

## コードベース分析の活用

`spec-researcher` エージェントのバックグラウンド分析結果（`context.md`）を活用して、**提案型のヒアリング**を行う。ゼロから聞くのではなく、分析結果を元に「今こうなっていますが、どうしますか？」と確認する。

### 活用パターン

**1. 既存パターンの確認（最も重要）**

分析結果から検出されたパターンをユーザーに提示し、継続するか変更するか確認する:

- 「プロジェクトを見ると、テストは `__tests__/` に置いていますね。この方針で続けますか？それともソースの隣に置く方がいいですか？」
- 「ディレクトリは機能別（`features/auth/`, `features/payments/`）で整理されていますね。新しい機能もこのパターンに合わせますか？」
- 「認証は JWT を使っていますね。今回の機能もこれを前提にしますか？」

**2. 技術スタックの活用**

ユーザーが技術的な話をしたとき、分析結果を元に具体的な選択肢を提示する:

- 「バリデーションですが、プロジェクトで既に Zod を使っていますね。これを使いますか？」
- 「状態管理は Pinia が入っていますね。新しい機能でも使いますか？」

**3. 不足の指摘（押しつけない）**

分析結果から「ないもの」に気づいたとき、自然に聞く:

- 「E2E テストはまだないようですが、今回の機能で必要そうですか？」
- 「エラーハンドリングの共通パターンが見当たらないのですが、何か決まったやり方はありますか？」

### タイミング

- **分析結果が返ってきたら**: 対話の区切りの良いタイミングで、分析から読み取れた規約を確認する。対話を中断して一方的に報告しない
- **対話が一区切りついたら**: まだ確認できていない規約について、分析結果を元に聞く
- **新規プロジェクト（分析結果が「コードベースなし」）の場合**: ベストプラクティスを提案して確認する形で聞く。「こだわりありますか？」ではなく「〜が一般的ですが、この方針でいいですか？」のように具体的な選択肢を提示する

---

## 終了時の処理

ユーザーが終了を告げたら:

### 1. hearing.md の最終更新

全ての未記録情報を `hearing.md` に反映する。

### 2. 全体サマリーの提示

ヒアリング内容の全体サマリーを構造化して提示する:

```
## ヒアリングサマリー

### やりたいこと
{1〜3 文の概要}

### 主なポイント
- {ポイント 1}
- {ポイント 2}
- ...

### まだ決まっていないこと
- {未決事項 1}
- {未決事項 2}

### 保存先
`.claude/works/{NNN}-{slug}/hearing.md`
```

### 3. 次のステップ案内

「ヒアリング内容は `.claude/works/{NNN}-{slug}/hearing.md` に保存しました。

次のステップ:
- `/spec-architect review .claude/works/{NNN}-{slug}/hearing.md` で仕様の深掘りとレビューに進めます
- `/spec-refiner continue` でヒアリングを再開することもできます」
