# レビュー観点チェックリスト

3つのレビューエージェントが使用するチェック項目リスト。各エージェントは担当セクションの観点を用いて仕様書を精査する。

---

## 完全性レビュー

仕様書に抜け漏れがないか、全ての振る舞いが定義されているかを検証する。

### ユースケース・機能要件

- [ ] 全ユースケースのハッピーパス（正常系）が定義されているか
- [ ] 各ユースケースのエラーケース（異常系）が網羅されているか
- [ ] エッジケース（境界値、空入力、最大値等）が明記されているか
- [ ] ユーザーの操作フロー（ステップバイステップ）が明確か
- [ ] 各機能の入力と出力が明確に定義されているか

### データ・状態管理

- [ ] 入力の境界値（最小値、最大値、許容長等）が明確か
- [ ] 状態遷移が全て定義されているか（状態遷移図または一覧）
- [ ] 各状態で許可される操作が明記されているか
- [ ] データのライフサイクル（作成・読取・更新・削除）が全て定義されているか
- [ ] データの初期状態とデフォルト値が明記されているか

### 非同期処理・外部連携

- [ ] 非同期処理の完了パスが定義されているか
- [ ] 非同期処理の失敗パス（タイムアウト、リトライ、補償処理）が定義されているか
- [ ] 外部システムとの連携仕様（API、プロトコル、データ形式）が明確か
- [ ] 外部システムの障害時の振る舞いが定義されているか

### 権限・認証

- [ ] ユーザーの役割（ロール）が定義されているか
- [ ] 各機能に対するアクセス権限が明記されているか
- [ ] 認証フロー（ログイン、ログアウト、セッション管理）が定義されているか
- [ ] 認可エラー時の振る舞いが定義されているか

---

## 一貫性レビュー

仕様書内で矛盾がないか、用語や概念が統一されているかを検証する。

### 用語・概念の統一

- [ ] 同じ概念に異なる名前が使われていないか（例: ユーザー/アカウント/メンバー）
- [ ] 用語集が存在し、全体で統一して使用されているか
- [ ] 略語が初出で定義されているか
- [ ] エンティティの名前が全セクションで一貫しているか

### データ形式・仕様の整合

- [ ] データ形式が統一されているか（日付形式、数値形式、文字コード等）
- [ ] 同じデータ項目の制約が全箇所で一致しているか（文字数制限等）
- [ ] API のリクエスト/レスポンス形式が統一されているか
- [ ] エラーコードの体系が統一されているか

### 方針・ルールの整合

- [ ] エラーハンドリング方針が全機能で統一されているか
- [ ] 命名規則が統一されているか（URL パス、パラメータ名、DB カラム名等）
- [ ] セキュリティ方針が全機能で統一されているか
- [ ] ページネーション、ソート等の共通パターンが統一されているか

### 参照整合性

- [ ] 他セクションへの参照が正しいか（存在しないセクションを参照していないか）
- [ ] 図表と本文の記述が一致しているか
- [ ] 前提条件として記載された内容が実際に定義されているか

---

## 実現可能性レビュー

技術的に実現可能か、パフォーマンス・セキュリティ上の問題がないかを検証する。

### 技術的実現可能性

- [ ] 現在の技術スタックで実現可能か
- [ ] 既存のアーキテクチャとの整合性があるか
- [ ] サードパーティ API やライブラリへの依存が明確か
- [ ] 依存する外部サービスの SLA や制約が考慮されているか
- [ ] マイグレーション計画が現実的か（既存データの移行等）

### パフォーマンス

- [ ] パフォーマンス要件（レスポンスタイム、スループット）が定義されているか
- [ ] 大量データ時の振る舞いが考慮されているか
- [ ] N+1 問題やバッチ処理の必要性が検討されているか
- [ ] キャッシュ戦略が必要な箇所が特定されているか

### セキュリティ

- [ ] 入力バリデーション方針が定義されているか
- [ ] SQL インジェクション、XSS 等の対策が考慮されているか
- [ ] 機密データの取り扱い方針が定義されているか（暗号化、マスキング等）
- [ ] CORS、CSP 等のセキュリティヘッダー方針が定義されているか
- [ ] レート制限が必要な API が特定されているか

### スケーラビリティ

- [ ] 想定ユーザー数・データ量が明記されているか
- [ ] スケーラビリティに問題のある設計パターンがないか
- [ ] 水平スケーリングが可能な設計か
- [ ] ボトルネックになりうる箇所が特定されているか

---

## 追跡可能性

仕様と設計の対応関係が追跡可能かを検証する。

- [ ] 各仕様要件に一意の識別子があるか
- [ ] 要件の優先度が定義されているか（Must/Should/Could/Won't）
- [ ] 要件間の依存関係が明確か
- [ ] 各要件が検証可能（テスト可能）な形で記述されているか
