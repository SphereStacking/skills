# レビューワークフロー詳細

仕様書レビューの進め方、エージェントへのプロンプトテンプレート、トリアージ基準、修正サイクルの手順を定義する。

---

## 1. レビュー開始前の準備

### 仕様書の前処理

1. 仕様書を読み込み、全体の構造を把握する
2. 仕様書が複数ファイルに分かれている場合、全ファイルを読み込む
3. 関連するコードベースのファイルを特定する（既存機能の変更の場合）
4. プロジェクトの技術スタックを把握する

### プロジェクトディレクトリの作成

```
.claude/spec-architect/{NNN}-{slug}/
├── spec.md              # 仕様書のコピー
├── review-report.md     # レビューレポート（レビュー後に生成）
└── design.md            # 詳細設計書（設計モード後に生成）
```

連番ルール: 既存の最大番号 + 1。ゼロパディング3桁（001, 002, ...）。

## 3. トリアージプロトコル

### 統合手順

1. 3つのエージェントの結果を収集する
2. 全指摘事項を1つのリストに統合する
3. 重複する指摘を統合する（異なるエージェントが同じ問題を検出した場合）
4. 深刻度の再評価:
   - 複数エージェントが同じ問題を指摘 → 深刻度を1段階引き上げ検討
   - 指摘の根拠が弱い場合 → 深刻度を1段階引き下げ検討

### 深刻度の判定基準

| 深刻度 | 影響範囲 | 発生可能性 | 例 |
|--------|---------|-----------|-----|
| **Critical** | システム全体 / データ | 高 | セキュリティホール、データ整合性の破壊、根本的な矛盾 |
| **High** | 主要機能 | 中〜高 | 重要なエッジケースの欠落、主要フローの未定義 |
| **Medium** | 補助機能 / UX | 中 | 用語の不統一、軽微な曖昧さ |
| **Low** | 体裁 / 読みやすさ | 低 | 表現の改善、書式の統一 |

## 4. 修正サイクル

### ユーザーへの提示

レビューレポートを生成後、以下の形式でユーザーに提示する:

```
## レビュー結果サマリー

- Critical: {N} 件
- High: {N} 件
- Medium: {N} 件
- Low: {N} 件

### 対応が必要な指摘（Critical/High）

{各指摘の概要と推奨対応を列挙}

### 修正を推奨する指摘（Medium）

{各指摘の概要を列挙}

Critical/High の指摘が全て解消されれば、設計フェーズに進めます。
```

### 修正後の再レビュー

1. ユーザーが仕様書を修正する
2. 修正箇所を特定する
3. 修正箇所に関連する観点のみ再レビューする（全体の再レビューは不要）
4. レビューレポートを更新する
5. Critical/High が 0 件になるまで繰り返す

### 再レビューの判断基準

- 修正が局所的（1セクション以内）→ 関連する観点のみ再チェック
- 修正が広範囲（複数セクション）→ 全3観点で再レビュー
- 新しいセクションが追加された → 全3観点で追加セクションのみレビュー

## 5. 設計書生成ワークフロー

### spec-analyst エージェントへのプロンプト構築

```
あなたはシニアソフトウェアアーキテクトです。以下の仕様書とレビュー結果に基づいて、
詳細設計書テンプレートの全セクションを埋めてください。

【仕様書】
{仕様書の全文}

【レビューレポート】
{レビューレポートの内容}

【設計書テンプレート】
{design-template.md の全文}

【技術スタック】
{プロジェクトの技術スタック}

【既存コードベースのパターン】
{関連する既存コードのパターン}

ガイドライン:
- テンプレートの全12セクションを必ず埋めること
- 各セクションのコメント（<!-- ガイドライン: ... -->）に従うこと
- Mermaid 図を積極的に活用すること
- 仕様トレーサビリティは 100% カバレッジを目指すこと
- 該当しないセクションは「該当なし」と理由を記載すること
```

### バリデーション後の修正

`validate_design.py` でエラーが検出された場合:
1. エラー内容を spec-analyst に伝える
2. 該当セクションのみ修正を依頼する
3. 再度バリデーションを実行する
4. 全エラーが解消されるまで繰り返す（最大3回）
5. 3回で解消しない場合、ユーザーに報告し手動修正を求める
